# こだわりポイント

## 工夫した点
1. CIを導入した
   - lint / test / type-check を初期段階で導入した
   - huskyとActionsも早期に導入し、コード品質を保つ基盤を整えた
   - test coverage測定も行い、品質の改善にも取り組んだ
2. UI/UXへの配慮
   - レスポンシブ対応、サーバーアクションとキャッシュの適切な利用を意識しUI/UXに配慮した
   - UIライブラリ(Ant Design)を導入し、実装コストを最低限にしつつ、一貫性のあるUIを実装した
   - ローディングスケルトンも実装した
3. issue / PRの活用
   - 実際の業務を想定し、issueの起票→実装→PRのサイクルを回して作業を行った

## ADR

### UIライブラリ/スタイリング
- 決定：Ant Design + CSS Modules + Sass
- 候補：shadcn / TailwindCSS
- 根拠：CSS Modules + Sassの組み合わせで、色やレスポンシブ周りの設定を共通化しつつ、純粋なCSSに近い記法で導入コストが低いため、今回のような小さなサービスに適切だと考えている。Ant Designはコンポーネント数が豊富でコンポーネントの自前実装のコストを最低限に減らせると考えた。また全て業務で使用しており基本的な利用方法の知見があったため、未経験のライブラリを使用することで発生する問題を回避したかった。

### コンポーネント設計
- 決定：Atomic Design
- 候補：feature-based / Container + Repository
- 根拠：粒度を明示することで責務の境界を統一し、再利用性を最大限活かすことができるため。業務でも使用しており、新しい設計に悩むコストを減らしたかった。

### テストライブラリ
- 決定：Vitest
- 候補：Jest
- 根拠：TSファーストで、実行速度の面でも有利なため採用。

### Linter/Formatter
- 決定：ESLint + Prettier
- 候補：Biome / oxlint
- 根拠：長年定番の組み合わせであり、今回のような小規模プロジェクトの場合は他候補の速度面での有利性は大きなメリットにならないと考えたため。

### バリデーション
- 決定：zod
- 候補：Yup
- 根拠：シンプルな記法と型推論能力が優れており、デファクトスタンダードなため。
